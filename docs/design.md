## Problem Statement

Street lighting fails during power outages. In Ukraine and similar grid-unstable regions, outages can last 4–12+ hours and recur nightly. Existing solar street lights each operate independently — when their individual battery depletes, they go dark. There is no coordination. A street with four solar lights can still go completely dark when all four batteries deplete simultaneously during a long outage.

The gap: **standalone solar lights + no coordination = no guarantee of coverage**. The fix: a swarm relay algorithm where lights with higher charge take over from those running low, ensuring at least one light at an intersection stays active as long as ANY light in the swarm has remaining capacity.

---

## What Makes This Cool

The "baton pass" demo: watch a live dashboard, cut power to the grid, and see Light #1 dim as its battery drops, Light #2 automatically ramp up as #1 signals it. The intersection never goes dark. The swarm is visibly alive. That's the "whoa" moment — distributed intelligence solving a life-safety problem, visible in real-time.

Secondary wow: 2+ days of autonomy for a single intersection, coordinated across 4 units, for roughly $7,000 in hardware. Cities pay more than that for a single generator that needs refueling.

1. **Power outages in the target market are frequent and long enough (4–12h) that 2-day battery autonomy is the right spec.** Evidence from research: EngoPlanet case data shows 2+ day autonomy is achievable; Nigeria/Ukraine context confirms extended outages are common. AGREE/DISAGREE?

## Product Definition (Full PRD)

### 1. Product Descriptions

**One sentence:** SolarSwarm is an autonomous solar street lighting system where lights communicate as a swarm to guarantee continuous coverage during power outages by relaying lighting duty based on battery level.

**One paragraph:** SolarSwarm replaces grid-dependent street lights with autonomous units — each with a solar panel, LiFePO4 battery, LED module, and a wireless chip. When the grid fails, units switch to battery mode. The swarm relay algorithm monitors all units in real-time: when any light's battery drops below 30%, it signals neighbors; at 10%, it transfers duty to the next unit in queue. An intersection with 4 SolarSwarm lights can stay lit for 2+ days even if the grid never returns, because the swarm ensures at least one unit is always active. A central dashboard gives city operators full visibility: battery levels, active light, uptime, and alerts.

**Investor pitch:** Urban streets in grid-unstable regions go dark during outages — increasing crime, accidents, and civilian risk. Existing solar lights solve daytime independence but fail on extended nights. SolarSwarm deploys a mesh-networked swarm where lights coordinate like a team: when one's battery drops, another takes over. We sell the hardware + software platform to municipalities on a hardware-as-service model. Target market: Ukraine (3+ years of deliberate grid attacks), sub-Saharan Africa (chronic underfunding), island nations (grid fragility). First customer: a city block pilot, then district, then city. The swarm coordination IP is the moat.

**ELI10:** Street lights that have their own solar battery inside, and they talk to each other through the air like walkie-talkies. When one light's battery is getting tired, it calls its neighbor: "Hey, my turn is almost over — you take over!" So the street always has light, even when the power is out for days, because the lights take turns helping each other.

### 7. User Journey

**Normal operation (grid up):**
1. Day: Solar panels charge LiFePO4 batteries via MPPT controllers
2. Evening: Grid powers the lights normally; batteries topped up
3. Dashboard shows: all units green, SOC 95–100%, grid: connected

**Outage begins:**
4. Grid voltage drops below threshold → each unit's controller detects it within 30 seconds
5. All units switch to battery mode simultaneously
6. Dashboard shows: "GRID FAILURE DETECTED — swarm mode active"
7. Unit priority queue established (by battery SOC, highest first)

**Swarm in action:**
8. Night hours 1–4: All units light at 80% brightness (battery conservation mode)
9. Hour 5: Unit #3 drops to 30% SOC → dims to 50% brightness, broadcasts "low battery" to swarm
10. Swarm controller: Unit #1 (highest SOC, 65%) ramps to 100%
11. Hour 7: Unit #3 drops to 10% SOC → goes to minimum (5% brightness), passes duty to Unit #2
12. Unit #3 signals "critical" → Unit #2 activates at 100%
13. Cycle repeats — baton passes from unit to unit as batteries deplete
14. Minimum: at least one unit active above 20% brightness at all times

**Emergency scenario:**
- If ALL units drop below 10% simultaneously: all units enter 5% minimum brightness mode (visibility maintained)
- Alert fired to operator: "SolarSwarm at [location]: critical — all units <10%"
- Operator dispatched or wait for sunrise recharge

**Communication-lost fallback (LoRa gateway failure):**
- If a unit does not receive a swarm heartbeat for >2 minutes, it enters solo mode
- Solo mode: unit lights at 70% brightness independently, ignoring swarm coordination
- On communication restore: unit re-announces its SOC, rejoins swarm, syncs queue
- This means swarm coordination degrades gracefully to N independent smart lights — no dark scenario even if LoRa fails entirely

**Grid restoration:**
15. Grid voltage restored → units detect within 60 seconds
16. All units switch back to grid power
17. Batteries begin charging via MPPT
18. Dashboard shows: "Grid restored — charging in progress"

### 8. System Architecture

```
[Solar Panel 200W]
        ↓
[MPPT Controller] ←→ [LiFePO4 Battery 48V/25Ah = 1.2kWh]
        ↓                        ↓
[LED Driver 50W] ←── [DC-DC Converter]
        ↓
[LED Module 50W, 130-170 lm/W]

[Microcontroller ESP32 or STM32]
  ├── Reads: Battery SOC (via coulomb counter / voltage divider)
  ├── Reads: Grid voltage (AC sense circuit)
  ├── Reads: PIR motion sensor (optional)
  ├── Controls: LED brightness via PWM → LED Driver
  └── Communicates via: LoRaWAN Module (SX1276 / RFM95)
              ↓
[LoRaWAN Gateway (1 per intersection, ~$150)]
              ↓
[The Things Network (TTN) or ChirpStack (self-hosted)]
              ↓
[Backend: FastAPI (Python) or Node.js]
  ├── Swarm orchestration logic (CENTRALIZED — backend decides relay order)
  ├── Battery SOC aggregation
  ├── Grid failure event processing
  └── REST API + WebSocket for dashboard
              ↓
[Dashboard: React + Recharts / or Grafana]
  ├── Live map of intersection (4 lights, color-coded by SOC)
  ├── Battery level chart (all units, time-series)
  ├── Active light indicator (which unit is "primary")
  ├── Uptime counter (hours without grid / hours of coverage maintained)
  └── Alert feed
              ↓
[Telegram Bot / Email alerts]
```

**Swarm topology decision: CENTRALIZED (backend orchestrates).** The backend receives SOC updates from all units every 30 seconds via LoRa, computes the relay queue (sorted by SOC descending), and sends brightness commands to each unit via LoRa downlink. Edge-only (decentralized peer-to-peer) is listed in the research as an option but requires more complex consensus logic — defer to v2. In the centralized model, gateway failure means units fall back to solo mode (see User Journey: comms-lost fallback).

**Priority queue assignment rule:** At the moment of grid failure, the backend ranks all units by current SOC (highest first). Unit #1 in the queue is the "primary" light (100% brightness). When primary drops below 30%, the next in queue ramps up. Queue is re-evaluated every 60 seconds — a unit that partially recharges during the day can re-enter at its new position.

**Which part creates most value:** The swarm relay algorithm + dashboard. The hardware is commodity (Fonroche, Signify make the same panels/batteries). The coordination logic and the operator visibility are the differentiators no competitor currently offers at the municipal level.

**Energy flow in numbers (from research Section 5, with overhead correction):**
- 4 × 50W LEDs, 10h/night = 2 kWh/night total if all run simultaneously
- With swarm relay (1 active at a time): 0.5 kWh/night LED load
- Parasitic drain (ESP32 microcontroller + LoRa module + MPPT idle + DC-DC converter losses): ~10–15% overhead → effective consumption ~0.57 kWh/night
- 1 × 200W panel, Kyiv climate (4 peak sun hours): 0.8 kWh/day generated (minus 10% MPPT+wiring losses = 0.72 kWh usable)
- Battery: 1.2 kWh per unit × 4 units = 4.8 kWh total swarm capacity
- Autonomy with overhead: 4.8 kWh ÷ 0.57 kWh/night = **~8.4 nights theoretical**
- Conservative 48-hour (2-night) KPI is well within margin even with overhead — confirmed feasible

### Technical Decisions Locked

**D2 — TTN integration method: HTTP Webhook (recommended) or MQTT subscription**

Both options:
- **HTTP Webhook** (recommended for hackathon): TTN sends a POST request to `POST /api/uplink` on FastAPI every time a unit sends a LoRa message. No persistent connection to manage. TTN free tier supports webhooks. Simplest async integration in FastAPI — just a route handler.
- **MQTT subscription**: FastAPI maintains a persistent MQTT connection to TTN's broker using `aiomqtt`. Slightly lower latency. Requires background task, reconnect logic, and a persistent connection to manage. Better for production; overkill for hackathon.

**Locked: HTTP Webhook for hackathon. MQTT for production/v2.**

---

**D3 — Simulation engine: separate Python process (chosen because hardware prototype is planned)**

3 simulated lights run as separate Python scripts that POST to `POST /api/uplink` every 30s, identical to what real hardware does via TTN webhook. When you add more physical units (Approach B full pilot), nothing changes in the backend — just stop running the simulation scripts.

Depletion math per tick (30s):
```
soc_per_tick = (0.57 kWh/night × 1/10h × 30s/3600s) / 1.2 kWh × 100%
             = (0.057 kW × 0.00833h) / 1.2 kWh × 100%
             ≈ 0.04% SOC per 30s tick (when unit is primary/active)
```
When unit is NOT primary (not lighting), drain is parasitic only: ~10% of the above = ~0.004% SOC per 30s tick.

---

**D4 — LoRa Class A accepted (30s max command latency)**

Class A devices open two downlink receive windows immediately after each uplink. At 30s uplink intervals, worst-case command latency is 30s. Under congestion, the device may miss the RX windows and wait until the next uplink cycle (up to 60s). The `<60s relay KPI` is met in normal conditions but may be borderline under RF congestion. Class C (always listening) would be instant but draws 10× more power — unacceptable for battery-powered operation.

**Documented limitation: relay KPI is 30–60s under congestion, not guaranteed <60s.**

---

**D5 — Dashboard state management: HTTP + WebSocket hybrid**

On page load: `GET /api/state` returns current SOC, brightness, grid status for all 4 units. Then subscribe to WebSocket `/ws` for live delta updates. On reconnect: re-fetch `GET /api/state` before subscribing. This ensures the dashboard never shows stale state after a browser refresh or network blip.

---

**D6 — LoRa packet format (5-byte binary)**

Compact binary to minimize airtime (EU868 1% duty cycle constraint):

```
Byte 0: unit_id       (uint8, 0–3)
Byte 1: soc_percent   (uint8, 0–100)
Byte 2: grid_status   (uint8, 0=fail, 1=ok)
Byte 3: battery_mv_hi (uint8, high byte of battery mV)
Byte 4: battery_mv_lo (uint8, low byte of battery mV)

Total: 5 bytes. Transmitted as base64 in TTN webhook payload field.
Decode: battery_mv = (byte[3] << 8) | byte[4]
```

Backend decodes from TTN webhook's `frm_payload` field (base64). Simulation scripts encode with the same format. Firmware encodes with the same format. All three agree.

---

```
  Server pushes on every state change:
  {"type": "state_update", "unit_id": 0, "soc": 71, "brightness": 100, "grid_status": 1}
  {"type": "swarm_event", "event": "relay_start", "from_unit": 0, "to_unit": 1}
  {"type": "alert", "level": "warning", "message": "Unit 0 SOC below 30%"}
  Used by: dashboard live updates
```

---

**D9 — Backend failure mode: heartbeat downlink added**

The comms-lost fallback (>2min no heartbeat → solo mode) was designed for gateway/LoRa failure. It does NOT cover backend process crash — units keep sending uplinks fine but no relay commands arrive.

Fix: backend sends a downlink heartbeat to each unit every 60s (every 2nd uplink cycle). Unit-side firmware: if no downlink received within 2 minutes, assume backend is dead, enter solo mode (70% brightness). This makes the comms-lost fallback work for both gateway failure AND backend failure.

```
Backend downlink heartbeat format (3 bytes):
  Byte 0: 0xFF (heartbeat marker)
  Byte 1: target_brightness (uint8, 0–100, current commanded brightness)
  Byte 2: swarm_active (uint8, 0=grid up/solo, 1=swarm mode)
```

---

**D10 — Database: SQLite (not PostgreSQL)**

At hackathon scale (~16 writes/minute, <1000 rows), PostgreSQL adds a separate Docker container and configuration overhead with no benefit. SQLite: single file, no container, built into Python via `aiosqlite`. Upgrade trigger: when scaling to >10 units or >1 deployment site, migrate to PostgreSQL + TimescaleDB.

Schema:
```sql
CREATE TABLE unit_state (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  unit_id INTEGER NOT NULL,
  soc_percent REAL NOT NULL,
  brightness INTEGER NOT NULL,
  grid_status INTEGER NOT NULL,
  battery_mv INTEGER,
  recorded_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_unit_time ON unit_state(unit_id, recorded_at);
```

### Failure Modes

| Codepath | Realistic failure | Test covers it? | Error handling? | User sees? |
|---|---|---|---|---|
| POST /api/uplink (binary decode) | Malformed 5-byte payload → IndexError | Yes — API test for bad payload | Should return 400 | Dashboard unchanged (silent backend error) — needs explicit 400 handler |
| SOC<30% relay trigger | Two units report SOC=29% simultaneously (race) | Unit test | asyncio event loop = single-threaded, no race | No issue — last write wins, queue re-evaluates |
| Heartbeat downlink | Backend sends downlink but TTN delivery fails (busy gateway) | E2E test | Unit stays in last-commanded state | May be dark if unit was dimmed pre-crash — **critical gap if backend crashes mid-relay** |
| Simulation script crash | One of 3 simulation processes dies | No test currently | Backend sees "no heartbeat" → after 2min, marks unit as solo | Dashboard shows unit as offline — acceptable demo outcome |
| SQLite write failure | Disk full | No test | None currently | Dashboard shows stale data silently — **critical gap: add try/except around DB writes** |
| WebSocket client disconnect | Browser closes mid-demo | E2E test | FastAPI WebSocket disconnect handler | No user impact — reconnect re-fetches state |

**Critical gaps:**
1. Backend crash mid-relay + TTN delivery failure → unit stays at dimmed brightness (not solo mode) until next uplink cycle. The heartbeat downlink (D9) mitigates but doesn't eliminate this.
2. SQLite write failure → silent stale data on dashboard. Add explicit error handling + WebSocket error event.
