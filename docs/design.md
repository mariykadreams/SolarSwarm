## Problem Statement

Street lighting fails during power outages. In Ukraine and similar grid-unstable regions, outages can last 4–12+ hours and recur nightly. Existing solar street lights each operate independently — when their individual battery depletes, they go dark. There is no coordination. A street with four solar lights can still go completely dark when all four batteries deplete simultaneously during a long outage.

The gap: **standalone solar lights + no coordination = no guarantee of coverage**. The fix: a swarm relay algorithm where lights with higher charge take over from those running low, ensuring at least one light at an intersection stays active as long as ANY light in the swarm has remaining capacity.

---

## What Makes This Cool

The "baton pass" demo: watch a live dashboard, cut power to the grid, and see Light #1 dim as its battery drops, Light #2 automatically ramp up as #1 signals it. The intersection never goes dark. The swarm is visibly alive. That's the "whoa" moment — distributed intelligence solving a life-safety problem, visible in real-time.

Secondary wow: 2+ days of autonomy for a single intersection, coordinated across 4 units, for roughly $7,000 in hardware. Cities pay more than that for a single generator that needs refueling.

1. **Power outages in the target market are frequent and long enough (4–12h) that 2-day battery autonomy is the right spec.** Evidence from research: EngoPlanet case data shows 2+ day autonomy is achievable; Nigeria/Ukraine context confirms extended outages are common. AGREE/DISAGREE?

## Product Definition (Full PRD)

### 1. Product Descriptions

**One sentence:** SolarSwarm is an autonomous solar street lighting system where lights communicate as a swarm to guarantee continuous coverage during power outages by relaying lighting duty based on battery level.

**One paragraph:** SolarSwarm replaces grid-dependent street lights with autonomous units — each with a solar panel, LiFePO4 battery, LED module, and a wireless chip. When the grid fails, units switch to battery mode. The swarm relay algorithm monitors all units in real-time: when any light's battery drops below 30%, it signals neighbors; at 10%, it transfers duty to the next unit in queue. An intersection with 4 SolarSwarm lights can stay lit for 2+ days even if the grid never returns, because the swarm ensures at least one unit is always active. A central dashboard gives city operators full visibility: battery levels, active light, uptime, and alerts.

Existing solar lights solve daytime independence but fail on extended nights. SolarSwarm deploys a mesh-networked swarm where lights coordinate like a team: when one's battery drops, another takes over. We sell the hardware + software platform to municipalities on a hardware-as-service model. Target market: Ukraine (3+ years of deliberate grid attacks), sub-Saharan Africa (chronic underfunding), island nations (grid fragility). First customer: a city block pilot, then district, then city. The swarm coordination IP is the moat.

**ELI10:** Street lights that have their own solar battery inside, and they talk to each other through the air like walkie-talkies. When one light's battery is getting tired, it calls its neighbor: "Hey, my turn is almost over — you take over!" So the street always has light, even when the power is out for days, because the lights take turns helping each other.

### 7. User Journey

*Written for Approach D demo. Real-world mapping noted in brackets.*

**Demo setup — "grid up" state:**
1. Both ESP32s powered via USB or battery, connected to demo laptop WiFi hotspot
2. 2 simulation scripts running on laptop (units #2 and #3)
3. All 4 units POSTing SOC to backend every 5s; dashboard shows 4 green bars (SOC ~95%)
4. LED strips on both physical ESP32s at 80% brightness (conservation mode baseline)
5. Dashboard shows: all units green, grid: connected

**Outage triggered — demo moment:**
6. Presenter covers photoresistor on unit #0 with hand [real world: grid voltage drops]
7. Unit #0 sends `grid_status=0` in next uplink (within 5 seconds)
8. Backend detects first outage signal → sets `swarm_active=true` → builds SOC priority queue (highest SOC = primary)
9. Dashboard shows: "OUTAGE DETECTED — Swarm mode active" event card; all unit indicators turn yellow
10. Units commanded: primary (#0 or highest SOC) → 100% brightness; others → 50% conservation

**Swarm in action — the baton pass (DEMO_SPEED=10: plays out in ~2 minutes on screen):**
11. SOC counters drain: primary runs at 100%, standby units at ~5% parasitic drain
12. Unit #0 SOC hits 30% → backend queues relay → next unit (#1) starts ramping up
13. Physical LED strip on unit #0 visibly dims; strip on unit #1 brightens — baton pass visible in the room
14. Dashboard event: "Relay started: Unit 0 → Unit 1 | SOC trigger: 29%"
15. Unit #0 SOC hits 10% → goes to 5% minimum; unit #1 becomes full primary (100%)
16. Dashboard event: "Relay complete: Unit 1 is primary"
17. Cycle repeats: units #1 → #2 (simulated) → #3 (simulated) as SOCs deplete in turn

**Emergency state — all units low:**
- When all 4 units <10% SOC: all LED strips hold at 5% minimum (still visible — not dark)
- Dashboard alert: "CRITICAL — All units below 10% — coverage at minimum"
- [Real world: operator dispatched or wait for sunrise recharge]
- Demo: uncover photoresistor to simulate sunrise/grid restore → triggers recovery

**Communication-lost fallback:**
- If ESP32 WiFi drops: HTTP POST times out after 3s
- After 3 consecutive timeouts: ESP32 holds last commanded brightness independently
- On reconnect: next successful POST returns fresh brightness command; unit rejoins swarm
- [Real world: same logic; LoRa loss → unit holds last state; on reconnect → re-syncs]

**Grid restored / demo reset:**
18. Presenter uncovers photoresistor → `grid_status=1` in next uplink
19. Backend detects all units report grid up → sets `swarm_active=false`
20. All units commanded back to 80% conservation brightness
21. SOC counters reset to 95% (software reset for next demo run)
22. Dashboard: "Grid restored — Units recharging" [real world: MPPT begins solar charging]

### 8. System Architecture

**Demo hardware stack (Approach D — what's actually being built):**

```
[Battery / USB power bank]
        ↓
[ESP32 (×2 physical units)]
  ├── Reads: Simulated SOC counter (software) — depleting each tick
  ├── Reads: Voltage divider on ADC pin → actual battery voltage (optional, resistors on hand)
  ├── Reads: Photoresistor on ADC pin → simulate solar charge when lit (optional)
  ├── Controls: LED strip brightness via PWM on GPIO (MOSFET gate if strip >3.3V)
  └── Communicates via: WiFi (built-in) → HTTP POST to backend every 5s
              ↓
[Backend: FastAPI (Python), runs on demo laptop]
  ├── Swarm orchestration logic (CENTRALIZED — backend decides relay order)
  ├── SOC aggregation: 2 real ESP32 payloads + 2 simulation script payloads
  ├── Sends brightness command back in HTTP response (no downlink complexity)
  └── REST API + WebSocket for dashboard
              ↓
[2× Simulation scripts (Python)]
  ├── Simulate units #2 and #3 with realistic SOC depletion curves
  └── POST to same /api/uplink endpoint every 5s
              ↓
[Dashboard: React + Recharts, runs on demo laptop]
  ├── Live grid of 4 lights (2 labeled "physical", 2 "simulated")
  ├── Battery SOC bars per unit (live drain animation)
  ├── Active light indicator + "baton pass" event feed
  └── Uptime counter: "X hours of coverage remaining"
```

**Production hardware stack (Approach B — city deployment reference):**

```
[Solar Panel 200W] → [MPPT Controller] ←→ [LiFePO4 48V/25Ah]
  → [LED Driver 50W] → [LED Module 50W]
[ESP32] → [SX1276 LoRa module] → [LoRaWAN Gateway] → [TTN] → [Backend]
```

**Swarm topology: CENTRALIZED (backend orchestrates).** Backend receives SOC from all 4 units every 5s, computes relay queue (SOC descending), returns brightness command in the HTTP response body. No separate downlink channel needed for demo. Edge peer-to-peer deferred to v2.

**Priority queue rule:** On outage trigger, backend ranks all 4 units by current SOC. Highest SOC = primary (100% brightness). At 30% SOC → next unit ramps up alongside. At 10% → full handoff, primary drops to 5%. Queue re-evaluated on every uplink tick.

**Demo wiring (per physical ESP32 unit):**
```
Battery/USB 5V → ESP32 VIN + GND

PWM brightness control:
  ESP32 GPIO (any PWM-capable, e.g. GPIO13)
    → 100Ω resistor (from resistors on hand)
    → N-channel MOSFET gate (e.g. 2N7000, IRF540, or similar)
  MOSFET Drain → LED strip negative terminal
  MOSFET Source → GND
  LED strip positive → 5V (USB) or 12V (if 12V strip + separate supply)

Optional battery voltage read:
  Battery+ → 10kΩ → GPIO34 (ADC) → 10kΩ → GND
  (voltage divider; adjust resistor ratio to keep ADC input ≤ 3.3V)

Optional photoresistor (outage trigger):
  3.3V → Photoresistor → GPIO35 (ADC) → 10kΩ → GND
  (high reading = bright/grid-up; low = dark/outage)
```

**Which part creates the most value:** The swarm relay algorithm + dashboard. Hardware is commodity — the coordination logic and real-time operator visibility are differentiators no competitor currently offers at municipal scale.

**Production energy math (reference only — not demo numbers):** 4× 50W LEDs with swarm relay (1 active at a time) = ~0.57 kWh/night effective. 4× 1.2 kWh LiFePO4 = 4.8 kWh total → ~8.4 nights theoretical autonomy. 48-hour KPI is conservative and feasible. See research Section 5 for full calculation.

### Technical Decisions Locked

**D2 — ESP32 communication: direct HTTP POST to backend (no TTN, no gateway)**

ESP32 posts JSON to `POST /api/uplink` on the FastAPI backend every 5 seconds over WiFi. Backend responds with the commanded brightness for that unit. No TTN, no MQTT, no LoRaWAN — just `WiFiClient` + `HTTPClient` on the ESP32 Arduino library. This is 20 lines of firmware code. When moving to production, swap the WiFiClient for a LoRa send and add TTN webhook handling on the backend — the `/api/uplink` contract is unchanged.

**Locked: Direct WiFi HTTP for demo. LoRaWAN + TTN webhook for production.**

---

**D3 — Simulation engine: separate Python process**

2 simulated units run as Python scripts that POST to `POST /api/uplink` every 5s, identical to what the real ESP32s do. When you add more physical units, stop running the corresponding simulation script. Nothing changes in the backend.

Depletion math per tick (5s) — scaled for demo speed (demo runs at 10× real speed so judges see the baton pass in ~2 minutes, not 5 hours):
```
soc_per_tick = demo_drain_rate (configurable, default 0.5% per 5s when primary/active)
             = 0.05% per 5s when unit is standby (parasitic only)
```
Set `DEMO_SPEED=1` for realistic rate (0.04%/5s active), `DEMO_SPEED=10` for fast demo.

---

**D4 — Command latency: <1s (WiFi replaces LoRa)**

HTTP round-trip on local WiFi: <100ms typical. Relay command is returned in the response body of each uplink POST — no separate downlink channel needed. The `<60s relay KPI` is easily met. No RF congestion, no duty cycle constraint, no Class A/C tradeoff.

---

**D5 — Dashboard state management: HTTP + WebSocket hybrid**

On page load: `GET /api/state` returns current SOC, brightness, grid status for all 4 units. Then subscribe to WebSocket `/ws` for live delta updates. On reconnect: re-fetch `GET /api/state` before subscribing. This ensures the dashboard never shows stale state after a browser refresh or network blip.

---

**D6 — Uplink payload format (JSON over WiFi)**

Plain JSON — no binary encoding needed, no duty cycle constraint, WiFi has no airtime cost:

```json
{
  "unit_id": 0,
  "soc_percent": 72,
  "grid_status": 1,
  "battery_mv": 3850,
  "type": "physical"
}
```

Backend parses with Pydantic model. Simulation scripts send identical JSON with `"type": "simulated"`. Firmware sends with `"type": "physical"`. All three agree. When moving to LoRa production, switch to 5-byte binary (see original D6 in git history) to respect EU868 duty cycle — but that's a firmware-only change, API contract stays the same.

---

**D7 — API contract (3 endpoints)**

```
POST /api/uplink
  Body: {"unit_id": 0, "soc_percent": 72, "grid_status": 1, "battery_mv": 3850, "type": "physical"}
  Response: {"brightness": 80, "swarm_active": false}
  Used by: ESP32 firmware (WiFi HTTP) AND simulation scripts — brightness command in response body

GET /api/state
  Response: {
```

**D9 — Backend failure mode: timeout on HTTP response**

With WiFi HTTP, the ESP32 detects backend failure immediately — the HTTP POST times out (set timeout to 3s). Firmware: if 3 consecutive POSTs time out (15s total), hold last brightness value and continue operating. No separate heartbeat channel needed — the uplink IS the heartbeat. For demo purposes this is sufficient; production LoRa system would use the 3-byte downlink heartbeat format described in the original architecture.

---

**D10 — Database: SQLite (not PostgreSQL)**

At hackathon scale (~16 writes/minute, <1000 rows), PostgreSQL adds a separate Docker container and configuration overhead with no benefit. SQLite: single file, no container, built into Python via `aiosqlite`. Upgrade trigger: when scaling to >10 units or >1 deployment site, migrate to PostgreSQL + TimescaleDB.

Schema:
```sql
CREATE TABLE unit_state (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  unit_id INTEGER NOT NULL,
  soc_percent REAL NOT NULL,
  brightness INTEGER NOT NULL,
  grid_status INTEGER NOT NULL,
  battery_mv INTEGER,
  recorded_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_unit_time ON unit_state(unit_id, recorded_at);
```

### Failure Modes

| Codepath | Realistic failure | Test covers it? | Error handling? | User sees? |
|---|---|---|---|---|
| POST /api/uplink (binary decode) | Malformed 5-byte payload → IndexError | Yes — API test for bad payload | Should return 400 | Dashboard unchanged (silent backend error) — needs explicit 400 handler |
| SOC<30% relay trigger | Two units report SOC=29% simultaneously (race) | Unit test | asyncio event loop = single-threaded, no race | No issue — last write wins, queue re-evaluates |
| Heartbeat downlink | Backend sends downlink but TTN delivery fails (busy gateway) | E2E test | Unit stays in last-commanded state | May be dark if unit was dimmed pre-crash — **critical gap if backend crashes mid-relay** |
| Simulation script crash | One of 3 simulation processes dies | No test currently | Backend sees "no heartbeat" → after 2min, marks unit as solo | Dashboard shows unit as offline — acceptable demo outcome |
| SQLite write failure | Disk full | No test | None currently | Dashboard shows stale data silently — **critical gap: add try/except around DB writes** |
| WebSocket client disconnect | Browser closes mid-demo | E2E test | FastAPI WebSocket disconnect handler | No user impact — reconnect re-fetches state |

**Critical gaps:**
1. Backend crash mid-relay + TTN delivery failure → unit stays at dimmed brightness (not solo mode) until next uplink cycle. The heartbeat downlink (D9) mitigates but doesn't eliminate this.
2. SQLite write failure → silent stale data on dashboard. Add explicit error handling + WebSocket error event.
