# SolarSwarm — MVP Architecture Plan (ESP-NOW Variant)

## Overview

Two ESP32 nodes communicate via ESP-NOW. The Master connects to WiFi and relays combined telemetry to the FastAPI backend every 5 seconds. The backend runs the swarm relay algorithm and returns brightness commands in the HTTP response. Commands for the Slave are relayed back via ESP-NOW. The dashboard receives live updates via WebSocket.

This is an update to the design doc (2026-06-27). The key change: the Slave no longer connects to WiFi. It communicates exclusively via ESP-NOW with the Master.

---

## 1. Responsibilities of Each ESP32

### ESP32 Node A — Master (Gateway)

- Connects to WiFi (demo laptop hotspot)
- Reads its own photoresistor → sets `grid_status` (bright=1/dark=0)
- Reads its own simulated SOC (counter depleting each tick)
- Controls its own LED strip via PWM
- ESP-NOW peer to Slave: receives Slave telemetry, sends brightness command back
- Aggregates own + Slave data into one combined HTTP POST every 5s
- Reads brightness commands from HTTP response body; applies own brightness
- Relays Slave's brightness command via ESP-NOW
- If 3 consecutive HTTP POSTs fail (3s timeout each): holds last brightness, retries next cycle
- If Slave hasn't sent data for >30s: marks slave as offline in next uplink payload

### ESP32 Node B — Slave (Sensor Node)

- NO WiFi — ESP-NOW only
- Reads its own photoresistor
- Reads its own simulated SOC
- Controls its own LED strip via PWM
- Sends telemetry struct to Master every 5s via ESP-NOW
- Receives brightness command from Master via ESP-NOW; applies to LED strip
- Fallback: if no ESP-NOW signal for >30s, holds last commanded brightness (does NOT go into auto mode)
- On reconnect: resumes receiving commands from Master

## 2. Data Each ESP32 Sends to the Backend

The Slave sends a compact struct to the Master via ESP-NOW (internal, not backend-facing):

```json
{
  "node_id": 1,
  "soc": 91,
  "light_level": 312,
  "brightness_actual": 60,
  "ts_ms": 1487240
}
```

The Master aggregates and sends one combined POST to the backend every 5s:

```json
{
  "nodes": [
    {
      "node_id": 0,
      "role": "master",
      "soc": 87,
      "light_level": 245,
      "brightness_actual": 100,
      "grid_status": 1,
      "rssi_wifi": -65,
      "online": true,
      "ts": "2026-06-28T10:00:00Z"
    },
    {
      "node_id": 1,
      "role": "slave",
      "soc": 91,
      "light_level": 312,
      "brightness_actual": 60,
      "grid_status": 1,
      "rssi_espnow": -72,
      "online": true,
      "ts": "2026-06-28T09:59:59Z",
      "slave_age_ms": 1200
    }
  ]
}
```

`slave_age_ms`: milliseconds since Master last received data from Slave. If >15000, backend marks node_1 as offline.

**Field note:** Use `soc` (not `battery_pct`) in the payload. The swarm relay algorithm reads `units[uid]["soc"]` — a field name mismatch is a runtime KeyError.

## 3. Commands the Backend Sends Back

The HTTP response to each POST returns brightness commands for both nodes:

```json
{
  "commands": [
    {
      "node_id": 0,
      "brightness": 100,
      "mode": "auto",
      "swarm_active": true
    },
    {
      "node_id": 1,
      "brightness": 0,
      "mode": "auto",
      "swarm_active": true
    }
  ]
}
```

In this example: Node 0 (soc=87%) is primary (brightness=100%), Node 1 (soc=91%) is actually higher — so Node 1 would be primary in the relay algorithm. The example above is illustrative only; actual values depend on the swarm engine output.

### Full Command Reference

| Field | Values | Description |
|-------|--------|-------------|
| `brightness` | 0–100 | LED strip PWM duty cycle |
| `mode` | `"auto"` / `"manual"` | Auto = follow swarm; manual = user override |
| `swarm_active` | bool | Whether swarm relay algorithm is running |
| `identify` | bool (optional) | Blink LED 3× for physical ID |

## 4. Backend Responsibilities

```
Backend API — FastAPI + SQLite, runs on demo laptop at localhost:8000
─────────────────────────────────────────────────────────────────────
POST /api/uplink
  Receive combined payload from Master
  → If slave_age_ms > 15000: mark node_1.online = false
  → Run swarm relay algorithm (compute_brightness)
  → Write to SQLite (with WAL mode enabled, wrapped in try/except)
  → Push WebSocket update to all dashboard clients
  → Return brightness commands for both nodes in response body

GET /api/state
  Full current state snapshot (dashboard uses on page load)
  Returns: all nodes with soc, brightness, online status, last_seen_s

GET /api/history
  Last 120 readings per node (10 minutes at 5s intervals)
  Used by SOC time-series chart

WebSocket /ws
  Push on every uplink: state_update, swarm_event, alert

POST /api/simulate/outage
  Set grid_status=0 on all nodes, activate swarm mode
POST /api/simulate/reset
  Reset all SOC to 95%, clear swarm mode
POST /api/simulate/speed
  Set DEMO_SPEED multiplier (1=realistic, 10=fast demo)
```

### Swarm Relay Algorithm (backend/swarm_engine.py)

```python
def compute_brightness(units: dict, swarm_active: bool) -> dict:
    """
    Input:  units = {node_id: {"soc": 87, ...}, ...}
    Output: {node_id: brightness_percent}
    """
    if not swarm_active:
        return {uid: 0 for uid in units}

    # Filter to online nodes only
    online = {uid: u for uid, u in units.items() if u.get("online", True)}
    if not online:
        return {uid: 0 for uid in units}

    # Sort by SOC descending; lower id breaks ties
    queue = sorted(online.items(), key=lambda x: (-x[1]["soc"], x[0]))

    # Emergency: all online nodes below 10%
    if all(u["soc"] <= 10 for _, u in queue):
        return {uid: 5 for uid in units}

    result = {uid: 0 for uid in units}
    primary_id, primary = queue[0]
    backup_id, backup = queue[1] if len(queue) > 1 else (None, None)

    if primary["soc"] > 30:
        result[primary_id] = 100
    elif primary["soc"] > 10:
        result[primary_id] = 50   # dimming — handoff in progress
        if backup_id:
            result[backup_id] = 50  # backup ramping up
    else:
        result[primary_id] = 5    # critical minimum
        if backup_id:
            result[backup_id] = 100  # backup now primary

    return result
```

### SQLite Schema and Setup

```sql
-- Add to db.py startup:
PRAGMA journal_mode=WAL;   -- prevents database is locked under concurrent reads

CREATE TABLE IF NOT EXISTS unit_state (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  node_id INTEGER NOT NULL,
  soc REAL NOT NULL,
  brightness INTEGER NOT NULL,
  grid_status INTEGER NOT NULL,
  recorded_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_unit_time ON unit_state(node_id, recorded_at);
```

All SQLite writes must be wrapped in try/except:

```python
try:
    await db.execute("INSERT INTO unit_state ...", values)
    await db.commit()
except Exception as e:
    logger.error(f"DB write failed: {e}")
    await ws_manager.broadcast({"type": "alert", "level": "warning",
                                 "message": "DB write failed — data may be delayed"})
```

```
Dashboard — React + Recharts, localhost:3000
──────────────────────────────────────────────────────────────────────
```

- `state_update` → update SOC bar + brightness
- `swarm_event` → add event to feed, shift active card glow
- `alert` → add alert to feed

## 6. Communication Flow

```
┌──────────────────────────────────────────────────────────────────────────┐
│                         COMMUNICATION FLOW                                │
└──────────────────────────────────────────────────────────────────────────┘

[ESP32 SLAVE — Node B]            [ESP32 MASTER — Node A]
 photoresistor                     photoresistor
 LED strip (PWM)                   LED strip (PWM)
 ESP-NOW peer                      ESP-NOW peer + WiFi client
      │                                   │
      │──── telemetry (every 5s) ────────►│
      │     ESP-NOW ~1-4ms                │
      │◄─── brightness cmd ───────────────│
      │     ESP-NOW ~1-4ms                │
                                          │
                              HTTP POST /api/uplink (every 5s)
                              body: combined payload for both nodes
                                          │
                                          ▼
                               ┌─────────────────────┐
                               │   BACKEND (FastAPI)  │
                               │   swarm_engine.py    │
                               │   db.py (SQLite+WAL) │
                               └──────────┬──────────┘
                                          │
                               HTTP response: brightness cmds
                               WebSocket: state_update, events
                                          │
                                          ▼
                               ┌─────────────────────┐
                               │  WEB DASHBOARD       │
                               │  React + Recharts    │
                               │  localhost:3000      │
                               └─────────────────────┘

Timing (normal operation):
  Slave → Master: ESP-NOW every 5s (~1-4ms latency)
  Master → Backend: HTTP POST every 5s (~50-100ms on local WiFi)
  Backend → Dashboard: WebSocket push on each uplink received

Command round-trip for Slave (worst case):
  Slave sends at t=0 → Master receives at t~0
  Master posts to backend at t=0-5s (next cycle)
  Response arrives at t=5s → Slave relay at t=5s
  Worst case: ~10s from Slave SOC change to Slave receiving new brightness
  At DEMO_SPEED=10 this is ~1s in demo time — acceptable
```

## Firmware: Key Implementation Notes

### Master (firmware/master/main.cpp)

Critical variables to configure before flashing:
```cpp
const char*  WIFI_SSID    = "YourHotspot";
const char*  WIFI_PASS    = "YourPassword";
const char*  BACKEND_URL  = "http://192.168.X.X:8000/api/uplink";
const int    UNIT_ID      = 0;
const int    LED_PIN      = 13;        // PWM-capable GPIO
const int    PHOTO_PIN    = 35;        // ADC input
const int    WIFI_CHANNEL = 6;         // Match your AP channel — CHECK THIS
const float  DRAIN_RATE   = 0.5;       // % SOC per 5s (0.5 realistic, 5.0 fast demo)
const uint8_t SLAVE_MAC[] = {0xAA, 0xBB, ...};  // Slave ESP32 MAC — find via Serial
```

ESP-NOW callback (runs in WiFi task, not true ISR — volatile struct is accepted risk):
```cpp
volatile struct SlaveData {
  int  soc;
  int  light_level;
  int  brightness_actual;
  long ts_ms;
  bool fresh;
} slave_data = {0};

void IRAM_ATTR onSlaveData(const uint8_t* mac, const uint8_t* data, int len) {
  memcpy((void*)&slave_data, data, min(len, (int)sizeof(slave_data)));
  slave_data.fresh = true;
}
```

Note: `volatile` without `portENTER_CRITICAL` is accepted risk for hackathon (5-min demo). A torn read would produce a garbage brightness flash — statistically unlikely but possible.

### Slave (firmware/slave/main.cpp)

Critical variables:
```cpp
const int    WIFI_CHANNEL  = 6;         // MUST match Master's AP channel
const uint8_t MASTER_MAC[] = {0xCC, 0xDD, ...};  // Master ESP32 MAC
const int    UNIT_ID       = 1;
const int    LED_PIN       = 13;
const int    PHOTO_PIN     = 35;
const float  DRAIN_RATE    = 0.5;       // MUST match firmware/master/main.cpp
```

Fallback behavior: if no ESP-NOW signal received for >30s, hold last commanded brightness. Do not enter photoresistor auto mode (would desync dashboard display from physical LED state).

### How to find ESP32 MAC addresses

1. Flash a minimal sketch that prints `WiFi.macAddress()` to Serial at 115200 baud
2. Open Arduino Serial Monitor
3. Note the MAC address (format: `AA:BB:CC:DD:EE:FF`)
4. Paste into the other ESP32's firmware as a byte array: `{0xAA, 0xBB, 0xCC, 0xDD, 0xEE, 0xFF}`
5. Do this for both ESP32s before the main firmware flash

## Known Accepted Risks

| Risk | Mitigation | Accepted because |
|------|-----------|-----------------|
| `volatile` struct without portENTER_CRITICAL | Statistically rare in 5-min demo | Adding critical section adds 4 lines; risk is acceptable for hackathon |
| Simulated nodes bypass ESP-NOW | Label them "simulated" on dashboard | By design; judges are told explicitly |
| No master reboot dedup | Backend detects slave_age_ms spike temporarily | Low probability in 5-min demo |
