## Key Interactions to Verify

### Swarm Algorithm (pytest — backend/tests/test_swarm_engine.py)
- Unit SOC drops to 29% → relay fires → next unit in priority queue ramps to 100%
- Unit SOC drops to 9% → duty fully transferred to next unit
- All units SOC < 10% → all enter emergency 5% brightness mode
- Grid failure event → priority queue initialized (units sorted by SOC descending)
- Priority queue re-evaluated every 60s → unit that partially recharged during day moves up
- Grid restoration → swarm exits, all units return to grid power

### Comms-Lost Fallback (pytest — backend/tests/test_comms_lost.py)
- Simulate no heartbeat for >2 minutes → unit enters solo mode (70% brightness)
- Simulate heartbeat resume → unit re-announces SOC, rejoins swarm, queue updated
- Confirm: comms loss does NOT turn off the light (stays at 70%)

### Simulation Engine (pytest — backend/tests/test_simulation.py)
- Verify SOC depletion rate: 0.57 kWh/night ÷ 1.2 kWh battery = ~17.4% SOC drop per night (50W LED × 10h = 0.5 kWh + 15% overhead)
- Verify tick rate: 30s tick decrements SOC by correct fraction
- Verify 8.4-night theoretical autonomy (4 units × 1.2 kWh ÷ 0.57 kWh/night) matches simulation output
- Verify simulated unit feeds into swarm engine identically to real unit

### API Endpoints (pytest — backend/tests/test_api.py)
- POST /api/uplink with valid 5-byte binary payload → unit state updated
- POST /api/uplink with malformed payload → 400 error returned
- GET /api/state → returns SOC, brightness, grid_status for all 4 units (1 real + 3 simulated)
- GET /api/state → active_unit field shows correct current primary
- WebSocket /ws → client receives state update within 5s of uplink

### WebSocket / Dashboard State (E2E — playwright or manual)
- Open dashboard → shows correct current state (not empty) via GET /api/state on mount
- Dashboard reconnects after disconnect → fetches fresh state, not stale cache
- State update arrives via WebSocket → dashboard updates without page refresh
- Simulated grid failure in backend → dashboard shows "GRID FAILURE — swarm mode active"

## Edge Cases

### Swarm Algorithm Edge Cases
- Unit with highest SOC goes offline mid-relay → next unit in queue takes over
- Two units have identical SOC at grid failure → deterministic tiebreak (unit_id ascending)
- Single unit remaining active at <10% → emergency mode, alert fires

### LoRa / Communication Edge Cases
- TTN delivers uplink with duplicate sequence number → deduplicate, don't double-count
- TTN webhook delivers uplink out of order (delayed packet) → state machine handles gracefully
- Backend receives uplink from unknown unit_id → log warning, ignore

### Simulation Edge Cases
- Simulation SOC reaches 0% → unit doesn't go negative; stays at 0%
- Simulation unit at 0% receives grid restoration → SOC starts recovering

### Dashboard Edge Cases
- Dashboard opened when all units in emergency mode → shows red alert immediately
- 4 dashboard tabs open simultaneously → all receive same WebSocket broadcast
- Dashboard opened with no units connected → shows "No units detected" not blank screen

## Critical Paths (end-to-end flows)

1. **Baton pass demo**: Grid power cut → backend detects grid_status=0 → activates swarm → physical unit reports SOC 72% (primary) → 30 minutes later SOC 29% → relay fires → simulated unit #2 ramps to 100% → dashboard shows transition — this must work live in the demo

2. **Comms-lost resilience**: Gateway powered off → after 2 minutes, all units in solo mode (70%) → gateway powered back on → units rejoin swarm → dashboard shows correct state

3. **Full-night simulation**: Start demo with all units at 100% SOC → simulate 10h of nighttime → verify relay chain fires in sequence → verify at least one unit stays active until SOC exhaustion
